# Гипотезы сценаристов — из внутреннего реестра кейсов (23.07.2026)

> Сгенерировано через `skill-hypothesis-generating.md`, Промпт 1 («Рынок/конкуренты»), но источник сигналов — **внутренний реестр кейсов команды** (`Cases_Scriptwriters_28_26.md`), а не внешний рынок. Формат и структура анализа идентичны стандартному прогону Промпта 1, но пометка «сила сигнала: сильный» здесь означает «есть внутренняя метрика и/или активная разработка», а не «конкурент подтвердил на своих клиентах» — это разные по надёжности типы валидации, не путать при дальнейшей приоритизации.
>
> **Нумерация:** в `Hypotheses_Scriptwriters_28_26.md` уже заняты H-SW-1…H-SW-16 (включая Vision scan Round 1 — H-SW-13…H-SW-16, проверка RegExp/utterances/impact-анализ/объяснение кода). Начинаем с **H-SW-17**, чтобы не было коллизий.

### H-SW-17 — Автотесты

> Сценаристы сократят время на написание автотестов (сейчас 9% спринта, дважды за спринт у каждого), если LLM будет автоматически генерировать автотесты по коду сценария на JAICP.

- **Источник:** Кейс А, `Cases_Scriptwriters_28_26.md`
- **Почему может работать:** структура автотестов простых сценариев примитивная; сценаристы сами детально расписали план обучения LLM (RAG + промпт); уже идёт пилот на Cursor именно по этому кейсу
- **Главный риск переноса:** JAICP — нишевая платформа, LLM вряд ли видела много её кода в обучении; качество без RAG/примеров непредсказуемо
- **Сила сигнала:** **Сильный** (есть метрика 9%/2 раза в спринт, приоритет 1 от самих сценаристов, уже в пилоте)

### H-SW-18 — Регулярные выражения

> Сценаристы сократят время на создание/оптимизацию regex (сейчас 5% спринта), если агент будет генерировать regex по описанию запроса и тестировать на реальной выгрузке клиентов.

- **Источник:** Кейс Б, `Cases_Scriptwriters_28_26.md`
- **Почему может работать:** regex — формальный язык, LLM традиционно хорошо с ним справляются; уже есть кастомный скрипт валидации (Вадима), который можно переиспользовать
- **Главный риск переноса:** интеграция агента со скриптом валидации Вадима не описана — если это отдельный ручной шаг, часть экономии теряется
- **Сила сигнала:** **Сильный** (5% трудоёмкости, ≥2 новых regex/релиз, ~20% сценаристов затронуты, приоритет 2)

### H-SW-19 — Ежедневная вычитка VOC

> Сценаристы сократят время на ежедневной вычитке VOC (сейчас 17% всего времени сценаристов), если агент сможет автоматически вычитывать диалоги и суммаризировать причины негатива.

- **Источник:** Кейс 2, `Cases_Scriptwriters_28_26.md`
- **Почему может работать:** команда уже инвестирует в это направление — 7 параллельных инициатив (агенты AF, JACP, JACP-Qwen, кросс-вычитка, суммаризация)
- **Главный риск переноса:** множество параллельных, несогласованных инициатив — риск не в технологии, а в распылении усилий и дублировании решений
- **Сила сигнала:** **Сильный** (17% времени — самая крупная метрика во всём реестре, уже в активной разработке)

### H-SW-20 — Кластеризация клиентских запросов

> Сценаристы сэкономят время на анализе клиентских запросов (доспрашивания/CatchAll), если агент сможет кластеризовать до 10 тыс. фраз и предлагать regex-паттерны.

- **Источник:** Кейс 5, `Cases_Scriptwriters_28_26.md`
- **Почему может работать:** кластеризация текста — зрелая задача (embeddings), MVP уже в разработке (этапы 5.1–5.3)
- **Главный риск переноса:** качество кластеризации на «грязных», разговорных клиентских фразах ниже, чем на чистом тексте; юзер-френдли интерфейс — отдельная инженерная задача
- **Сила сигнала:** **Сильный** (конкретный объём — 10 тыс. фраз, уже в разработке)

### H-SW-21 — Описание релизов

> Сценарист сэкономит 0.5–2 часа на PR, если ИИ будет автоматически анализировать изменённые файлы и генерировать тезисное описание релиза по направлениям/сегментам.

- **Источник:** Кейс 10, `Cases_Scriptwriters_28_26.md`
- **Почему может работать:** это фактически diff-суммаризация (анализ разницы между версиями кода и пересказ своими словами) — зрелая задача, аналог автогенерации changelog
- **Главный риск переноса:** качество зависит от того, ясна ли бизнес-цель правки из самого кода — LLM может не понять «зачем», а не только «что»
- **Сила сигнала:** **Сильный** (единственный кейс с явным диапазоном времени на конкретное действие — 0.5–2ч/PR)

### H-SW-22 — Новости БЗ

> Ответственный сценарист сэкономит время на ежедневном скрининге и распределении новостей БЗ, если агент будет фильтровать релевантные новости и определять подходящий интент/тематику.

- **Источник:** Кейс 3, `Cases_Scriptwriters_28_26.md`
- **Почему может работать:** процесс алгоритмизируем — классификация/фильтрация текста
- **Главный риск переноса:** «экспертная фильтрация» опирается на неформализованное доменное знание — не описано, по каким явным критериям эксперт принимает решение
- **Сила сигнала:** **Средний** — детальный процесс описан, но метрики времени/частоты нет — **требует проверки**

### H-SW-23 — Качество задач в Jira

> Ответственный сценарист сократит время на ежедневную оценку ~1500 задач в Jira, если ИИ будет проверять их на соответствие регламентирующим документам.

- **Источник:** Кейс 4, `Cases_Scriptwriters_28_26.md`
- **Почему может работать:** «соответствие регламенту» — это набор формальных правил, LLM подходит для такой проверки
- **Главный риск переноса:** нет данных, сколько времени реально уходит на 1500 задач — неясно, какая часть эффекта достижима
- **Сила сигнала:** **Средний** — объём есть, времени/частоты нет — **требует проверки**

### H-SW-24 — Реестр интентов

> Сценаристы сэкономят время на поддержке актуальности реестра интентов (~8 тыс. строк), если агент будет автоматически сверять изменения из гита и подсвечивать несостыковки.

- **Источник:** Кейс 8, `Cases_Scriptwriters_28_26.md`
- **Почему может работать:** механическая сверка данных (сравнение кода и таблицы), не творческая генерация
- **Главный риск переноса:** целевая админка уже запланирована на 4кв2026 — риск потратить ресурсы на временное решение, которое скоро станет ненужным
- **Сила сигнала:** **Средний** — объём есть (8к строк), времени на поддержку нет — **требует проверки**

### H-SW-25 — Дайджест релизов

> РН сценаристов сэкономит время на сборе информации для дайджеста релизов, если агент суммаризирует релизные задачи/PR/Confluence в готовый дайджест.

- **Источник:** Кейс 6, `Cases_Scriptwriters_28_26.md`
- **Почему может работать:** суммаризация из нескольких источников (задачи, git, Confluence) — RAG + суммаризация текста
- **Главный риск переноса:** «иногда приходится уточнять у разработчиков» — если ключевая информация не задокументирована, агент не восстановит её без того же уточнения у людей
- **Сила сигнала:** **Слабый** — нет метрики вообще, только описание процесса — **требует проверки**
