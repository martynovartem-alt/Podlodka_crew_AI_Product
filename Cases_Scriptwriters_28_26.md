## Кейс 3 — Автоматизация новостей БЗ

**Часть 1 — ознакомление и распределение новостей одним ответственным сценаристом**
- Полностью ручной процесс, включает:
  - ежедневное ознакомление со всеми поступившими новостями (открывает БЗ, читает все новости)
  - экспертная фильтрация — что применимо к ЧБ, а что нет (например, новость по оформлению драфта для оператора не берётся в работу)
  - экспертная оценка на наличие подходящего интента и тематики в ЧБ
  - передача новости в работу ответственному сценаристу в личные сообщения
  - внесение новости в реестр (Google Docs)

**Часть 2 — работа с переданными кейсами владельцем сценария**
1. Сценарист, которому прислали новость, заводит задачу «Новости БЗ»
2. После анализа новости — в течение часа отписаться в файле новостей, берётся ли новость в работу; если нет — указать причину в столбце «Берётся в работу / нет — почему не берётся»
3. Закрасить строку новости одним из трёх цветов:
   - красный — новость не берётся в работу
   - жёлтый — нужно уточнить доп. информацию или посмотреть спрос
   - зелёный — новость берётся в работу
4. Если несколько сценаристов отмечены по одной новости — цветом заливает последний отписавшийся; если хотя бы один вносил изменения на прод — строка заливается зелёным
5. Заполнить столбец «Дата прода» — дата, когда изменения планируются на проде
6. Заполнить столбец «Ссылка на задачу в Jira»
7. Столбец «Проверено качеством» остаётся незаполненным
8. В своей задаче Jira «Новости БЗ» внести всю информацию по поступившим новостям (независимо от того, берётся новость в работу или нет)
9. Если по новости ранее уже была задача от заказчика — в файл новостей вносится задача от заказчика, а не новая

## Кейс 8 — Оптимизация интентов new

- Проблематика: список интентов с бизнесовым названием, путями до стейтов в коде, статусом «в бою/не в бою», данными по единому классификатору и прочей информацией хранится в объёмной Google-таблице (~8 тыс. строк, лист «Стейт_интент_NEW10»). Работать с ней крайне затруднительно, в том числе поддерживать актуальность. Важно, чтобы новые стейты/интенты заносились туда своевременно, а удалённые с прода переводились в статус «не в бою».
- Идеи:
  - перенести информацию из файла и синхронизировать с гитом, чтобы он подтягивал изменения

- Контекст: целевое решение (админка вместо Google-файла) будет разрабатываться не раньше конца 4 кв. 2026 — требуется автоматизировать ведение записей об интентах раньше

## Кейсы с AS IS / TO BE и метриками (доп. реестр, передан 23.07.2026)

> Отдельный, более детальный реестр — с описанием текущего процесса (AS IS), видения сценаристов (TO BE), трудоёмкостью в % от рабочего времени, частотой использования и приоритетом. Это сильно ценнее для гипотез, чем первый реестр — здесь уже есть количественные данные и самостоятельно расставленные сценаристами приоритеты.

### Кейс А — Написание автотестов

**AS IS:** Автотесты прописываются вручную при изменении текущего кода и при создании нового — тесты для фраз и для event. Если в коде есть проверки/функции/версионность, тесты пишутся с их учётом на каждую ситуацию (например, если в коде два ответа в зависимости от версии приложения в одном сценарии — тест пишется дважды с учётом версионности).

**TO BE (видение сценаристов):**
1. Обучить LLM (при помощи RAG или иначе) понимать принцип построения автотестов — структура автотестов простых сценариев достаточно простая; возможно, получится и со сложными
2. Обучить LLM синтаксису JAICP — тоже довольно примитивный, можно обучить на примере сценария с комментариями. Возможно, уже есть модели, понимающие JAICP-код без RAG
3. Обернуть всё в промпт вида: *РОЛЬ: тестировщик сценариев на платформе JAICP. ЦЕЛЬ: написание автотестов для покрытия 100% кода сценария. ИНСТРУКЦИЯ: 1) изучи файл sample.sc — пример кода сценария с комментариями; 2) изучи файл sample.xml — пример кода автотестов с комментариями; 3) используя документацию, напиши автотесты для сценария из файла.*

**Трудоёмкость:** 9% в каждый спринт
**Частота использования:** 2 раза в спринт, используется каждым сценаристом при каждой раскатке
**Комментарий:** **Приоритет 1.** Ожидаемый эффект: (1) большее покрытие кода автотестами, (2) сокращение времени на написание автотестов.

### Кейс Б — Создание регулярных выражений

**AS IS — создание нового regex:**
1. Подготовка списка запросов, которые должны быть охвачены регулярным выражением
2. Создание регулярного выражения
3. Тестовая проверка отработки запросов на regex101.com
4. Проверка на реальной выгрузке клиентов — с помощью кастомного скрипта Вадима

**AS IS — оптимизация текущих regex:**
1. При выявлении не охваченных regex фраз (на вычитке) или при необходимости добавить новые фразы из-за изменения продукта — корректировка regex
2. Тестовая проверка на regex101.com
3. Проверка на реальной выгрузке клиентов (скрипт Вадима) + тест фраз, по которым проводилась оптимизация

**TO BE (видение сценаристов):** Агент, способный создавать регулярные выражения под нужный запрос и тестировать его.

**Трудоёмкость:** 5% в каждый спринт
**Частота использования:** по мере необходимости — экспертно ~20% сценаристов в каждый релиз проводят манипуляции с текущими regex; минимум 2 новых regex создаётся в каждый релиз
**Комментарий:** **Приоритет 2.** Ожидаемый эффект: (1) сокращение времени на написание regex, (2) улучшение классификации за счёт качества regex.

5. Агент-кодер пишет логику в соответствии с отредактированными ответами и группировкой от агента-аналитика
6. Агент-кодер пишет 100%-покрытие кода автотестами; сценарист проверяет листинг тестов на соответствие ожидаемой работе сценария

**Трудоёмкость:** 25% за спринт создания
**Частота использования:** низкочастотный кейс — создание новых сценариев редкое, в среднем 2–3 сценария за квартал
**Комментарий:** Приоритет низкий, по остаточному принципу.

1. Определение необходимости правок — изменения в продукте, необходимость расширения логики сценария из-за новых запросов и пр.
2. Проработка новой логики сценария с учётом изменений, корректировка блок-схемы под новую логику
3. Изменение в классификаторе под потребности новой логики — пополнение трейна, добавление/удаление сущностей
4. Правка кода под новую логику, правка ответов (предварительное составление и согласование)
5. Правка автотестов под новый код
6. Для интеграционных сценариев — взаимодействие и проверка со стороны команды развития; может потребоваться отчёт о тестировании в зависимости от вида правок

**TO BE:** не применимо — вероятно, совпадает с юзкейсом «Простые сценарии» (В), отдельное видение не формировалось.

**Трудоёмкость:** 6% в спринт
**Частота использования:** регулярно — но здесь бóльшую роль играет сама аналитическая работа и продумывание логики до правок в код, а не само редактирование кода
**Комментарий:** Приоритет низкий, доп. инструмент. Видится, что кейсы В–Д могут использовать/переиспользовать инструменты друг друга — расписан полный цикл, затрагивающий весь пул работ по сценарию, однако в зависимости от конкретной ситуации и задачи часть пунктов может не требоваться.

### Сводная таблица (кейсы с метриками)

| Кейс | Трудоёмкость | Частота | Приоритет (сам сценарист) |
|---|---|---|---|
| А — Автотесты | 9% в спринт | 2 раза в спринт, у каждого сценариста | **1** |
| Б — Регулярные выражения | 5% в спринт | ~20% сценаристов в релиз; ≥2 новых regex за релиз | **2** |
| В — Простые сценарии | 25% за спринт создания | 2–3 сценария/квартал | Низкий |
| Г — Сложные сценарии | 80% за спринт создания | 2–3 сценария/квартал | Низкий |
| Д — Правки кода сценариев | 6% в спринт | Регулярно | Низкий, доп. инструмент |

**Наблюдение:** сами сценаристы уже расставили приоритеты — автотесты (А) и regex (Б) явно приоритетнее написания сценариев (В, Г) и правок (Д), несмотря на то что В/Г показывают куда более высокую % трудоёмкость. Это логично: В/Г — редкие события (2-3 раза в квартал), тогда как А происходит **у каждого сценариста дважды за спринт** — F (частота) здесь явно перевешивает T (трудоёмкость на случай) при выборе приоритета, ровно то же самое рассуждение T×F, которое использовалось для гипотез аналитиков.

## Пилотный план: тестирование Cursor по кейсам А–Д (май–июнь)

> План пилота вайб-кодинг инструмента (Cursor, бесплатная версия) на копии репозитория — по всем пяти кейсам из раздела выше (А–Д). Ответственный за организацию пилота — Стребков Леонид; по каждому кейсу назначен отдельный исполнитель. Колонки с датами (25.05–28.06) есть в исходном плане, но конкретные отметки о выполнении по датам не заполнены — только структура этапов.

### Этап 1 — Подготовка и настройка Cursor (бесплатная версия)

*Ответственный: Стребков Леонид*

1. Скачать и установить Cursor (бесплатную версию) каждому члену команды
2. Локально клонировать копию актуальной мастер-ветки проекта
3. Подготовить краткий `cursorrules` — системный промпт на уровне проекта (особенности платформы JAICP, синтаксис, принятые соглашения по именованию, примеры типовых конструкций сценариев)
4. Проверить корректность работы автодополнения и чата (Ctrl+L / Ctrl+K)
5. Протестировать простые запросы

### Этап 2 — Обучение

*Ответственный: Стребков Леонид*

1. **Мини-воркшоп 1: Базовый Cursor** — управление контекстом (файлы, символы), промпты в чате (Ctrl+L), инлайн-редактирование (Ctrl+K), Command Palette, работа с терминалом
2. **Мини-воркшоп 2: Приёмы для сценаристов** — как просить ИИ написать тестовый кейс для бота, как сгенерировать сложное регулярное выражение, как по ТЗ создать заготовку простого/сложного сценария
3. **Мини-воркшоп 3: Безопасная доработка кода** — как с помощью Ctrl+K менять конкретные блоки без поломки общей логики, как заставить ИИ объяснить предложенный код

### Этап 3 — Пилотный проект на копии репозитория

По одному исполнителю на кейс — соответствуют кейсам А–Д из раздела выше:

**3.1 Автотесты** — *Трошин Дмитрий (РБ)*
1. Выбрать сложный сценарий из проекта РБ, чтобы ИИ самостоятельно написал автотесты
2. Проверить сгенерированный тест
3. Внести корректировки

**3.2 Регулярные выражения** — *Чаплыгин Антон (РБ)*
1. Сформировать ТЗ на естественном языке
2. Получить regexp
3. Провалидировать на тестовых строках в данном проекте

**3.3 Простые сценарии** — *Чайка Данил (СМЕ)*
1. Написать промпт: «Создай сценарий по дивидендам и пропиши автотесты к нему»
2. Использовать инлайн-генерацию (Ctrl+K → вставляем описание, ИИ выдаёт код)
3. Добавить специфичные интеграции вручную

**3.4 Сложные сценарии** — *Ослоповский Максим (РБ)*
1. Написать промпт, чтобы ИИ из блок-схемы сформировал сценарий, оформил автотесты, указал динамику и учёл возраст клиентов
2. Провалидировать сгенерированный сценарий

**3.5 Правки кода** — *Приленский Александр (АИ)*
1. Выделить нужный участок, дать новые вводные по сценарию, чтобы ИИ учёл это во всех необходимых файлах и внёс изменения
2. Провалидировать сгенерированный результат

*Ответственный: Стребков Леонид*

### Связь с уже описанными кейсами

Пилот напрямую тестирует TO BE-видение кейсов А–Д из раздела выше, но через готовый инструмент (Cursor + промпт-инжиниринг), а не через кастомного агента с нуля — это дешёвый способ проверить, работает ли направление (LLM понимает JAICP-синтаксис и генерирует адекватный код/автотесты/regex), прежде чем инвестировать в разработку выделенных агентов (агент-аналитик/редактор/кодер из TO BE кейса В).

**Важно для трактовки результатов пилота (см. DL-2 в decision-log.md):** вайб-кодинг ранее уже проваливался на контекстно-тяжёлом legacy-коде именно чат-ботовых сценариев — «поймать ошибку инструмента всё равно требует понимания кода». Если пилот покажет те же симптомы (сгенерированный код выглядит правдоподобно, но содержит тонкие ошибки, которые сценарист не сразу замечает) — это будет прямым повторением причины провала DL-2, а не новым результатом.
